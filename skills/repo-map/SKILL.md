# repo-map: agent-friendly repository link-map

Produce a single markdown file, **`MAP.md`** at the repo root, that lets an agent
(or human) understand a codebase from one read: a short **overview** + a **link-map**
of modules with one-line responsibilities and who-uses-what links. It is built from
a deterministic scan ([scripts/scan.py](scripts/scan.py) = the muscle) that the model
turns into prose (the intelligence).

## What it is — and is NOT (read first)

- **It is a derived, regenerable, code-coupled artifact.** It lives **in the repo**
  and is **regenerated** when the code changes — it has no durable value once stale.
- **It is NOT brain knowledge.** Do not put `MAP.md` (or the scan JSON) into an
  engram brain — the brain holds durable human thinking, not code-derived indexes
  (that is engram's explicit boundary). Keep the map next to the code.
- **Boundary vs engram (one line)**: *repo-map* answers "what is this codebase and
  how is it wired, right now" (derived, in-repo); *engram* answers "what I decided/
  learned worth revisiting" (durable, in-brain). The map is a great **input** for
  writing a brain `projects/<repo>/overview.md`, but the two stay separate artifacts.

## Usage

```
/repo-map               -- generate or refresh MAP.md for the current repo (default)
/repo-map status        -- run the scan and show the skeleton without writing MAP.md
/repo-map to-brain      -- distill MAP.md into a short brain project overview (needs engram)
```

## Workflow — generate / refresh

1. **Scan (muscle)** — run the deterministic scanner from the repo root:
   ```bash
   python "<skill_dir>/scripts/scan.py" --root . --json
   ```
   Installed as a plugin: `${CLAUDE_PLUGIN_ROOT}/skills/repo-map/scripts/scan.py`.
   It returns JSON: `stack`, `languages`, `entrypoints`, `modules` (path · file/LOC
   counts · langs), `internal_edges` (module→module, weighted), `declared_dependencies`,
   and `observed_external`. `<skill_dir>` is the directory holding this SKILL.md.

2. **Sample (intelligence)** — read the highest-signal files to learn *intent*, not
   just structure: the README, each `entrypoint`, the top manifests, and the
   `README`/index of the largest few `modules`. Do not read everything — the scan
   already gives the skeleton; read enough to write accurate one-liners.

3. **Write `MAP.md`** at the repo root with these sections:
   - **Overview** — purpose in ~2 lines, stack, and **entry points** (where to start
     reading, as links).
   - **Module map** — one row per significant module: a relative link to it, a
     one-line responsibility, and its key internal dependencies (`→ [other-module]`,
     as links). Order by importance (size/centrality from `internal_edges`).
   - **External dependencies** — the handful that actually shape the design (from
     `observed_external`/`declared_dependencies`), one line each on why.
   - **Footer** — a line noting the map is generated by repo-map and regenerable, so
     readers do not hand-edit it.
   Weave the cross-module links **into the prose/table** (brain-style), not as a dump.
   Keep it to one screenful per ~20 modules; collapse trivial leaf dirs.

4. **Report** — path written, module/edge counts, and offer `to-brain`.

## Workflow — refresh (re-run)

The map is derived, so **regenerate, don't merge**: re-run the scan and rewrite
`MAP.md` wholesale. There is nothing to preserve (unlike a brain). If a `CLAUDE.md`
exists, suggest a one-line pointer ("Agents: read `MAP.md` first") so the map is
actually used. For automatic refresh, the user can re-run `/repo-map` before
releases or wire it into a post-commit/CI step (optional, their call).

## Workflow — `to-brain` (the engram pipe)

Only when the user asks. Read `MAP.md`, distill the **durable** takeaways (purpose,
architecture-at-a-glance, key design decisions, notable gotchas) into a short
overview, and hand it to the **engram** skill to place at `projects/<repo>/overview.md`
in the brain (linked + MOC-updated). Do **not** copy the whole map — only the
durable essence. The map stays in the repo; the distilled overview goes to the brain.

## Rules

1. **One file, repo root**: default output is `MAP.md` at the repo root. Honor an
   existing convention if the project/`CLAUDE.md` specifies a docs location.
2. **Regenerable, never hand-merged**: rewrite wholesale on refresh; never treat it
   as a source of truth to preserve.
3. **Never into the brain**: do not write the map or scan JSON under an engram base.
   Cross to the brain only via `to-brain`, distilled.
4. **Links are the point**: express module relationships as real markdown links so an
   agent can jump. Avoid a flat file dump.
5. **Altitude over completeness**: map modules (directories), not every file; the goal
   is navigation, not an exhaustive index. Collapse trivial dirs.
6. **User language**: write `MAP.md` and reports in the user's language.
