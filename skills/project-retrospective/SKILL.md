# Project Retrospective — 경력 프로필용 개발 이력 추출

## 개요

현재 브랜치의 git 커밋 히스토리에서 사용자가 지정한 기능/주제 관련 커밋을 분석하여,
이력서에 바로 활용할 수 있는 추상화된 경험 데이터 문서를 생성한다.

## 입력

사용자가 기능명(feature topic)을 지정한다. 예시:
- "인증/인가 미들웨어"
- "감사 로그 시스템"
- "커스텀 DB 드라이버"
- "AI 기반 개발 자동화 스킬"

기능명이 불명확하면 사용자에게 확인 후 진행한다.

## 실행 단계

### Step 1: 커밋 히스토리 수집

관련 커밋을 최대한 넓게 수집한다. 아래 방법을 조합:

1. **파일 경로 기반**: 해당 기능의 주요 디렉토리/파일 경로를 추정하여 `git log -- <path>` 실행
2. **키워드 기반**: 기능과 관련된 키워드로 `git log --grep` 실행 (한국어/영어 모두)
3. **전체 히스토리 스캔**: 위 결과가 부족하면 `git log --oneline --reverse` 전체를 스캔하여 관련 커밋 필터링

수집된 커밋에서 아래 정보를 추출:
- 커밋 해시, 날짜, 메시지
- 주요 커밋의 `git show --stat` (변경 파일/규모 파악)
- 아키텍처 전환점이 되는 커밋의 상세 diff

### Step 2: 패턴 분석

수집된 커밋에서 아래 패턴을 식별:
- **설계 결정**: 왜 이 방식을 선택했는가 (대안은 무엇이었는가)
- **진화 과정**: 구조가 바뀐 지점 (v1 → v2 → v3)
- **문제 해결**: 버그/장애 → 시도한 해결책들 → 최종 결정
- **리팩토링**: 왜 기존 구조를 바꿨는가
- **삽질/디버깅**: 여러 커밋에 걸친 시행착오 흔적

**제외 대상**: 뻔한 CRUD, 단순 문서 수정, chore, 오타 수정, import 정리

### Step 3: 문서 작성

아래 구조를 **반드시** 준수한다.

#### 파일 위치 및 이름
- 경로: `para/resources/reviews/project-retrospective/`
- 파일명: 내용에 맞는 영문 kebab-case `.md`

#### Front Matter
```yaml
---
title: {행위 중심의 한 줄 제목}
created: {오늘 날짜}
purpose: 경력 프로필 작성용 원본 데이터
period: {커밋 날짜 범위, 예: "약 2개월"}
---
```

#### 본문 구조

```markdown
# {제목 — 행위 중심, 한 줄}
  예: "폐쇄망 키 관리 시스템의 인증/인가 미들웨어 체인을 설계하고 3차례 진화시킨 경험"

## 배경
{왜 이 작업이 필요했는가. 3줄 이내.}

---

## 핵심 흐름

### 1. {흐름 제목}

**문제 인식**: {무엇이 문제였는가}

**판단**: {왜 이 방향을 선택했는가. 대안 대비 이유.}

**행동**: {구체적으로 무엇을 했는가}

**결과**: {어떤 성과가 나왔는가}

---

### 2. {다음 흐름}
...반복...

---

## 반복적으로 등장한 판단 기준
1. **{기준명}** — {설명}
2. ...

---

## STAR 요약

### S (Situation)
{상황 설명, 3~4줄}

### T (Task)
{과제 정의, 2~3줄}

### A (Action)
- {핵심 행동 1}
- {핵심 행동 2}
- ...

### R (Result)
- {핵심 성과 1}
- {핵심 성과 2}
- ...
```

## 작성 규칙 (중요)

이 규칙들은 출력 품질의 핵심이다. 반드시 지킨다.

### 추상화 규칙
출력 문서에 아래 항목을 **절대 포함하지 않는다**:
- 코드명, 함수명, 변수명 (예: `withAuth`, `writeError`, `AuthContext`)
- 파일 경로 (예: `internal/server/middleware.go`)
- 커밋 해시 (예: `bc96c5d`)
- 구체적 라인 수 (예: "586줄→153줄" 대신 "75% 감소")

대신 아래처럼 추상화한다:
- "인증 검증 미들웨어" (withAuth 대신)
- "에러 응답 함수의 인자를 7개에서 5개로 단순화" (writeError 시그니처 대신)
- "서버 모듈의 코드량이 75% 감소" (줄 수 대신 비율)

### 내용 규칙
- 각 흐름은 반드시 **문제 인식 → 판단 → 행동 → 결과** 4단계 구조
- "판단" 단계가 가장 중요 — 왜 이 방향을 선택했는지, 대안은 왜 기각했는지
- 핵심 흐름은 3~6개가 적정. 너무 적으면 깊이 부족, 너무 많으면 산만
- "반복적으로 등장한 판단 기준"은 여러 흐름에서 공통으로 나타난 의사결정 패턴
- STAR 요약은 이력서에 복붙할 수 있을 정도로 완결적이어야 함

### 톤 규칙
- 같은 디렉토리에 기존 문서가 있으면 톤과 깊이를 맞춘다
- 기술적 정확성을 유지하되, 비개발자 면접관도 구조를 이해할 수 있는 수준
- 과장하지 않는다 — 사실 기반, 커밋에서 확인 가능한 내용만

## 참고

기존 문서 예시가 필요하면 `para/resources/reviews/project-retrospective/` 디렉토리의
기존 파일을 참조하여 톤과 깊이를 맞춘다.
