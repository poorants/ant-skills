# Work Report — 사내 임원 보고용 작업내역서 생성

## 개요

현재 브랜치의 git 커밋 히스토리에서 사용자가 지정한 기간 또는 기능 범위의 작업을
분석하여, 비개발자(임원/관리자)에게 제출할 수 있는 작업내역서를 생성한다.

## 입력

사용자가 아래 중 하나를 지정한다:
- **기간**: "이번 주", "3월 1주차", "2월 작업분"
- **기능**: "인증 관련", "사용자 관리 API", "DB 드라이버"
- **둘 다**: "3월에 한 인증 작업"

불명확하면 사용자에게 확인 후 진행한다.

## 실행 단계

### Step 1: 커밋 수집

지정 범위의 커밋을 수집한다.

- 기간 지정 시: `git log --after="날짜" --before="날짜"` 사용
- 기능 지정 시: 관련 경로/키워드로 필터링
- 수집 후 chore, 오타 수정, import 정리 등 사소한 커밋은 제외

### Step 2: 작업 묶음 식별

개별 커밋을 의미 단위로 묶는다. 하나의 "작업"은 여러 커밋에 걸칠 수 있다.

분류 기준:
- 같은 기능/화면을 다루는 커밋끼리 묶기
- 리팩토링/개선 작업은 별도 묶음
- 버그 수정은 해당 기능 묶음에 포함하거나, 독립 건이면 분리

### Step 3: 문서 작성

#### 파일 위치 및 이름
- 경로: `para/resources/reviews/`
- 파일명: `work-report-{날짜 또는 주제}.md`

#### 문서 구조

아래 구조를 따른다. 내용의 길이와 상세도는 작업 규모에 맞게 조절한다.

```markdown
# {작업 제목}

## 1. 개요

{이번 작업이 무엇인지 2~3줄로 요약.
 전체 작업의 맥락과 범위를 비개발자가 이해할 수 있게 설명.}

## 2. 주요 작업 내용

### {작업 묶음 1 제목}

{무엇을 했는지 3~5줄. bullet point 또는 서술형.}

### {작업 묶음 2 제목}

{무엇을 했는지 3~5줄.}

...필요한 만큼 반복...

## 3. 정리

{전체 작업의 의미를 1~2줄로 요약.
 단순 나열이 아니라 "이 작업들이 합쳐져서 어떤 가치를 만드는지" 관점.}
```

#### 선택적 섹션 (작업 성격에 따라 포함)

작업 내용에 따라 아래 섹션을 추가할 수 있다. 모든 섹션을 넣을 필요는 없고,
해당 작업에 의미 있는 것만 선택한다.

- **API 목록 테이블**: API 개발 작업인 경우. `| API | 기능 |` 형태
- **기능 묶음 설명**: API가 많을 때 영역별로 그룹핑하여 설명
- **보안/인증 방식**: 보안 관련 작업일 때 비개발 관점으로 풀어서 설명
- **적용 범위**: 변경이 영향을 미치는 시스템 영역 나열
- **기대 효과**: 이 작업으로 얻게 되는 운영/관리 측면의 이점
- **확인 가능한 위치**: 코드 경로 (선택적, 검증 필요 시에만)

## 작성 규칙 (중요)

### 톤과 수준
- **비개발자가 읽는 문서**다. 개발 용어는 꼭 필요할 때만 쓴다.
- API 경로(`POST /api/v1/auth/login`)는 작업 범위를 명확히 할 때 사용 가능.
  단, 코드 내부 구조(함수명, 패키지명, 변수명)는 쓰지 않는다.
- "구현했다", "개발했다" 대신 **"구성했다", "정리했다", "연결했다", "마련했다"** 같은
  비기술적 동사를 선호한다.
- 기술 개념을 써야 할 때는 바로 뒤에 쉬운 설명을 붙인다.
  예: "OTP 2차 인증(로그인 시 추가 확인 단계)"

### 분량
- **간결하게**. 임원이 5분 안에 읽을 수 있는 분량.
- 작업 묶음당 3~5줄이면 충분하다. 길어야 좋은 게 아니다.
- API 목록 테이블은 있으면 좋지만, 3개 이하면 테이블 없이 본문에 녹인다.

### 구조
- "한 줄 정리" 또는 "정리" 섹션으로 마무리하여 전체 작업의 의미를 압축한다.
- 작업 순서(시간순)보다 **기능 단위**로 묶는 것이 읽기 좋다.
- 번호 매기기는 대섹션만. 소항목은 bullet이나 서술로 처리.

### 하지 말 것
- 커밋 해시, 브랜치명 언급하지 않는다.
- 리팩토링/디버깅 과정의 시행착오를 상세히 쓰지 않는다 (결과만).
- "~했습니다" 경어체 사용하지 않는다. "~했다" 서술체 통일.
- 과장하지 않는다. 사실 기반으로 담백하게.

## 참고

기존 작업내역서가 `para/resources/reviews/` 에 있으면 톤과 깊이를 맞춘다.
